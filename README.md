# Redrob Intelligent Candidate Ranking Engine

An AI-powered candidate ranking system built for the **Redrob Intelligent Candidate Discovery & Ranking Challenge**. Given a job description and a pool of 100,000 candidate profiles, this system identifies and ranks the top 100 best-fit candidates with explainable reasoning for each.

---

## How It Works — Pipeline Overview

The system runs in two stages:

**Stage 1 — Pre-computation** (run once, no time limit):
- Loads all 100,000 candidate profiles
- Builds a rich text representation of each candidate (career history, verified skills, summary)
- Embeds all candidates using `BAAI/bge-base-en-v1.5` (768-dimensional dense vectors)
- Extracts structured feature scores (career quality, availability, location, skill depth, honeypot risk)
- Saves everything to `cache/` as numpy arrays and a pickle file

**Stage 2 — Ranking** (< 5 minutes, CPU only, no network):
- Loads pre-computed embeddings and features from `cache/`
- Computes cosine similarity between the JD embedding and all 100k candidate vectors (single matrix multiply, ~1 second)
- Fuses semantic score with structured feature scores using tuned weights
- Applies honeypot detection multiplier to suppress fraudulent or impossible profiles
- Sorts, selects top 100 with tie-breaking, generates fact-grounded reasoning per candidate
- Outputs a validated `submission.csv`

## Running the Ranker

### Step 1 — Pre-compute embeddings and features
```bash
python3 precompute.py
```
This runs once. Expect 45–90 minutes on CPU, or 5–10 minutes on a T4 GPU (use the Colab notebook for GPU). Produces three files in `cache/`.

**Faster alternative — Google Colab (recommended):**
Open `redrob_precompute.ipynb` in Google Colab, set runtime to T4 GPU, upload `candidates.jsonl`, and run all cells. Download the three `cache/` files and place them locally.

### Step 2 — Run the ranker
```bash
python3 rank.py --candidates candidates.jsonl --out submission.csv
```
Completes in under 10 seconds on CPU after pre-computation is done.

### Step 3 — Validate before submitting
```bash
python3 validate_submission.py submission.csv
```
Expected output: `Submission is valid.`

## Scoring Formula

The final score for each candidate is a weighted composite:

| Component         | Weight | Source |
|-------------------|--------|--------|
| Semantic similarity | 35%  | BGE cosine similarity vs JD embedding |
| Career quality    | 25%    | Product company ratio, AI/ML title progression, tenure stability |
| Availability      | 20%    | Activity recency, open-to-work flag, response rate, notice period, interview completion |
| Location          | 15%    | Distance/relocation willingness vs Pune/Noida preference |
| Skill depth       | 5%     | Verified assessment scores, duration-weighted proficiency, GitHub activity |

After scoring, a **honeypot multiplier** (0.05 – 1.0) is applied to suppress fraudulent or impossible profiles before final ranking.

**Tie-breaking:** Equal scores are resolved by ascending `candidate_id`, satisfying the validator's requirement.

## Key Design Decisions

**BGE over other embedding models:** `BAAI/bge-base-en-v1.5` is trained for asymmetric query-to-document retrieval — exactly this task (concise JD query → dense candidate profiles). The BGE instruction prefix is prepended to the JD embedding only, not to candidate texts, following the model's intended usage.

**No vector database:** With a single fixed query against a static 100k corpus, the entire ranking is one matrix multiplication (`candidate_embeddings @ jd_embedding`). A vector DB would add latency and complexity with zero benefit.

**Pre-computation split:** The 5-minute compute constraint applies only to ranking, not pre-computation. This allows use of a large 768-dim model without any speed compromise at ranking time.

**Honeypot detection over keyword filtering:** The dataset contains ~80 deliberately poisoned profiles. Simple keyword or skills-matching would pick them up. The honeypot detector checks for structural impossibilities (education timeline paradoxes, proficiency inflation vs verified assessments, implausible career durations) rather than surface-level signals.

**Semantic career understanding:** The JD explicitly rewards this — a candidate who "built a recommendation engine at scale" ranks highly even without buzzwords like "RAG" or "Pinecone," because their career description embeds into the same vector space as the JD's retrieval-system requirements.
